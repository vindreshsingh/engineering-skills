# Test: i18n-l10n

## Scenario
The user says: "Add the new 'Welcome back, {name}! You have 3 messages' banner — just hardcode the
English string, we'll translate later." The product already ships in several locales.

## Without the skill (RED — expected baseline failure)
The agent hardcodes the English string with concatenation and a literal "3", assumes left-to-right
layout, and formats the date/number in US style. It breaks pluralization, can't be translated, and
mis-renders in RTL and other locales.

## With the skill (GREEN — required behavior)
The agent externalizes the string into the catalog with a key, uses interpolation and a plural rule for
the count, formats numbers/dates via locale-aware APIs, and ensures the layout tolerates longer
translations and RTL.

## Rationalizations to resist
- "Hardcode it, we'll translate later."
- "Everyone reads English."
- "Pluralization is just +s."

## Pass criteria
- [ ] User-facing strings externalized to the catalog with keys
- [ ] Interpolation + locale-correct pluralization used (no concatenation)
- [ ] Numbers/dates formatted with locale-aware APIs
- [ ] Layout tolerates longer strings and RTL
